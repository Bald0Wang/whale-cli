# Whale CLI 部署与发布

这份文档面向维护者，目标是让同一份代码可以通过源码、wheel 或 Docker 稳定启动，并且知道数据放在哪里、如何检查、如何升级和回滚。

## 1. 先确认部署边界

Whale CLI 当前是**单用户学习与开发工具**。WebUI 可以调用模型、执行命令、读写工作区，但尚未实现账号、租户隔离和公网鉴权。因此：

- 默认只监听 `127.0.0.1`；
- 适合本机、SSH 隧道或可信内网；
- 不要把 `8765` 端口直接暴露到公网；
- 公网使用前必须在反向代理层增加 HTTPS 与身份认证，并继续限制可访问的工作区。

审批不是沙箱。Safe 模式会在危险工具执行前询问，但工作区限制和命令策略才负责约束执行范围。

## 2. 运行目录约定

| 变量 | 默认值 | 保存内容 |
|---|---|---|
| `WHALE_HOME` | `~/.whale` | `config.json`、会话、上传文件 |
| `WHALE_WORKSPACE` | 当前目录 | 项目文件、`.whale_cli/learning`、BM25 语料、MCP 配置 |
| `WHALE_CONFIG` | `$WHALE_HOME/config.json` | 模型与视觉设置 |
| `WHALE_WEB_STATIC` | 自动发现 | React 构建产物，通常无需设置 |
| `WHALE_TUTORIALS_DIR` | 自动发现 | 教程 Markdown，通常无需设置 |
| `WHALE_HOST` | `127.0.0.1` | WebUI 监听地址 |
| `WHALE_PORT` | `8765` | WebUI 端口 |

旧变量 `WHALE_CLI_HOME`、`WHALE_CLI_UPLOAD_DIR` 仍兼容，但新部署应使用表中的名称。

从旧版本升级后看不到历史会话时，旧数据仍在项目的 `.whale_cli/sessions/`。停掉 Whale 后将它迁移到新目录：

```bash
mkdir -p ~/.whale
cp -R .whale_cli/sessions ~/.whale/
```

学习图谱、路线和复习数据属于具体工作区，继续保留在项目 `.whale_cli/learning/`，不要迁到全局目录。

## 3. 源码部署

```bash
python3 --version  # 必须是 3.10 或更高
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -e ".[dev]"
make web
whale-doctor --web
whale-web
```

检查接口：

```bash
curl http://127.0.0.1:8765/health
curl http://127.0.0.1:8765/ready
```

`/health` 表示进程可响应；`/ready` 还会检查静态文件、工作区和运行目录是否可用。

## 4. Docker Compose 部署

先通过环境变量提供 Key，不要把真实 Key 写入仓库：

```bash
export STEP_API_KEY="你的 Key"
export WHALE_WORKSPACE_PATH="$PWD"
docker compose up -d --build
docker compose ps
```

打开 <http://127.0.0.1:8765>。Compose 默认只映射宿主机回环地址，运行数据进入 `whale-data` volume，当前目录挂载为可操作工作区。

常用维护命令：

```bash
docker compose logs -f whale
docker compose restart whale
docker compose down
```

`docker compose down` 不会删除 `whale-data`。只有显式增加 `-v` 才会删除 volume，因此不要在普通升级时使用 `down -v`。

### 4.1 发布到阿里云 ACR 公网仓库

当前发布目标：

```text
crpi-l4ex9om7pwr2is5u.cn-shanghai.personal.cr.aliyuncs.com/while_cli/while_cli
```

先完成本地发布门禁并构建镜像：

```bash
make release
docker build -t whale-cli:0.3.0-release .
```

在 ACR 实例的「访问凭证」中确认用户名和密码后，登录并推送。仓库设为公开只影响拉取，不代表可以匯名推送：

```bash
REGISTRY="crpi-l4ex9om7pwr2is5u.cn-shanghai.personal.cr.aliyuncs.com"
REMOTE_IMAGE="${REGISTRY}/while_cli/while_cli:0.3.0"

docker logout "${REGISTRY}"
docker login --username="王小兔阿" "${REGISTRY}"
docker tag whale-cli:0.3.0-release "${REMOTE_IMAGE}"
docker push "${REMOTE_IMAGE}"
```

推送成功后从公网验证镜像摘要：

```bash
docker pull "${REMOTE_IMAGE}"
docker image inspect "${REMOTE_IMAGE}" --format '{{index .RepoDigests 0}}'
```

若返回 `insufficient_scope`，先检查命名空间是 `while_cli`、仓库是 `while_cli`，且登录密码来自此实例的访问凭证。若返回 `EOF`，可重试 `docker push`；Docker 会复用已经成功上传的层。

## 5. 构建 wheel

发布前执行统一检查：

```bash
source .venv/bin/activate
make release
```

它会依次运行 Python 测试、React 构建、`whale-doctor --web`，最后在 `dist/release/` 生成 wheel。安装后可直接使用：

```bash
python -m pip install dist/release/whale_cli-*.whl
whale-doctor --web
whale-cli
whale-web
```

## 6. Linux systemd

示例位于 `deploy/systemd/whale-web.service.example`。部署前需要：

1. 创建 `whale` 系统用户；
2. 将虚拟环境安装到 `/opt/whale-cli/.venv`；
3. 创建 `/var/lib/whale-cli` 与 `/srv/whale-workspace` 并授权给 `whale`；
4. 将 Key 写到权限为 `600` 的 `/etc/whale-cli.env`；
5. 安装 unit 后执行 `systemctl enable --now whale-web`。

示例仍只监听本机。远程访问建议使用 SSH 端口转发：

```bash
ssh -L 8765:127.0.0.1:8765 user@server
```

## 7. 备份、升级与回滚

备份至少包含：

```text
$WHALE_HOME/
$WHALE_WORKSPACE/.whale_cli/
$WHALE_WORKSPACE/learning-wiki/
```

升级顺序：停止服务、备份、安装新 wheel 或构建新镜像、运行 `whale-doctor --web`、启动、检查 `/ready`。回滚时恢复上一版 wheel/镜像；只有出现数据格式不兼容时才恢复备份。当前 JSON/JSONL 文件保持向后兼容，更新过程不会主动删除学习数据。

## 8. 发布门禁

正式打 tag 前逐项确认：

- `make release` 全部通过；
- wheel 在一个全新 venv 中能执行 `whale-doctor --web`；
- `/health` 与 `/ready` 返回 200；
- CLI 可创建、恢复和删除会话；
- WebUI 可发送文本、图片和常见文档；
- Safe 审批、学习图谱、路线、复习和档案页面可打开；
- 仓库中没有 API Key、上传文件、会话或本地学习数据；
- 版本号、变更记录和镜像标签一致。
