# Whale CLI 教程小节标准化写作模板

这份文件用于指挥能力较弱的模型修改 Whale CLI 教程。它规定每章怎么展开、每类小节写什么、代码和结论如何核对。执行模型不需要重新设计结构，只需按模板补全事实。

适用范围：`docs/新手入门/00` 至 `19`，以及以后新增的教程章节。

## 1. 使用方法

给执行模型提供以下输入：

| 输入 | 示例 |
|---|---|
| 目标教程 | `docs/新手入门/04-AgentLoopv0-从聊天到会做事的循环.md` |
| 对应源码 | `src/whale_cli/soul/soul.py`、`src/whale_cli/soul/toolset.py` |
| 对应测试 | `tests/test_soul_integration.py` |
| 本章新增机制 | Agent Loop |
| 前一章已知概念 | LLMClient、messages |
| 本章不讲内容 | 工具具体实现、并行执行、生产级恢复 |

执行顺序固定为：

1. 读取教程全文。
2. 读取对应源码，找到真实入口、调用者和返回值。
3. 读取对应测试，确认哪些行为已经被验证。
4. 列出正文与代码不一致之处。
5. 按本模板修改教程。
6. 运行对应测试和文档检查。

不能只看旧教程改写。旧教程可能已经过时，源码和测试才是当前事实依据。

## 2. 写作底线

### 2.1 只写能核对的事实

每个功能判断必须落入以下四类之一：

| 标签 | 什么时候使用 | 写法 |
|---|---|---|
| 当前实现 | 源码中存在，并能通过测试或命令验证 | “当前 `Soul.run()` 会在每轮模型调用前检查上下文长度。” |
| 教学简化 | 为了讲清概念，示例省略了保护或工程细节 | “下面的示例省略消息持久化，只展示循环主干。” |
| 扩展练习 | 当前仓库没有，留给读者实现 | “练习：为失败工具调用增加重试计数。” |
| 生产差距 | 成熟系统通常需要，但项目尚未实现 | “当前没有跨进程恢复，CLI 退出后定时 Loop 不会继续。” |

禁止把“建议实现”“生产系统通常有”写成 Whale CLI 已有功能。

### 2.2 首次出现的名词必须当场解释

首次介绍术语时，依次写四件事：

1. 中文白话定义。
2. 它接收什么输入。
3. 它改变什么状态或产生什么输出。
4. 它在 Whale CLI 的哪个文件中实现。

示例：

> Toolset 是工具注册表。模型给出工具名和 JSON 参数后，Toolset 找到对应 Tool，执行审批与 Hook，再返回统一结果。正式实现位于 `src/whale_cli/soul/toolset.py`。

不要用一个陌生术语解释另一个陌生术语。

### 2.3 先给最小心智模型，再给完整实现

知识介绍固定分两层：

- 第一层只解释这一机制最小需要什么，建议使用输入、处理、输出三步。
- 第二层映射 Whale CLI 的真实模块、状态和失败路径。

如果一上来就贴 80 行源码，初学者只能看到语法，不能看出机制。

### 2.4 代码必须和正文相互解释

代码块前写“为什么看这段”，代码块后写“运行时发生了什么”。不能连续堆放代码，也不能只逐行翻译语法。

每段代码只承担一个教学目的。推荐 10 至 35 行；超过 50 行时拆成多个步骤，或只展示关键函数并链接正式文件。

### 2.5 使用自然、直接的中文

- 直接陈述事实，删除“众所周知”“值得注意的是”“至关重要”等填充词。
- 少用“这不仅是……更是……”“从……到……”等宣传式句型。
- 不把所有内容强行分成三点。
- 不使用表情符号装饰标题。
- 粗体只用于很短的风险提示，不要每段都加粗。
- 中文段落通常写 2 至 5 句，长短句可以交替。
- 同一名词保持同一写法，不为避免重复而频繁换同义词。

## 3. 一章内容怎么安排

每章按下面顺序组织。特殊章节可以删掉不适用的小节，但不能打乱学习顺序。

| 顺序 | 小节 | 读者要得到什么 | 必须包含 |
|---|---|---|---|
| 1 | 开场 | 知道本章解决哪个具体问题 | 失败场景、本章新增机制 |
| 2 | 本章目标 | 知道学完如何验收 | 2 至 4 条可观察结果 |
| 3 | 前置知识 | 知道哪些概念默认已掌握 | 最多 3 个前章链接 |
| 4 | 最小模型 | 先看懂机制，不被工程代码淹没 | 输入、处理、输出、停止条件 |
| 5 | 工作原理 | 看懂一次完整运行过程 | 调用顺序、状态变化、失败分支 |
| 6 | 真实实现 | 能从教程跳到仓库代码 | 文件、入口、调用者、返回值 |
| 7 | 设计原理 | 知道为什么这样拆模块 | 选择理由、替代方案、代价 |
| 8 | 配置或用法 | 能亲手运行 | 完整命令、示例输入、预期现象 |
| 9 | 错误与边界 | 知道失败时先查什么 | 常见错误、诊断入口、安全边界 |
| 10 | 模块化代码 | 能独立复现本章机制 | 可运行快照、正式实现映射 |
| 11 | 测试 | 能验证代码而非相信正文 | 精确命令、测试函数、关键断言 |
| 12 | 动手实验 | 能主动改变输入观察结果 | 正常用例、失败用例 |
| 13 | 自测与小结 | 能复述核心机制 | 3 个问题、已实现与未实现 |
| 14 | 下一章 | 知道下一章只增加什么 | 一句话说明增量 |

## 4. 每个小节的通用写法

任何讲解型小节都先按下面五步写。某一步不适用时可以省略，但不要改变顺序。

### 4.1 先说本节解决什么

开头用一两句话给出具体问题。问题必须能在运行时观察，不能是空泛目标。

推荐：

> 用户输入 `/clear` 后，旧消息不能继续进入下一轮模型请求，但旧会话仍要保留在磁盘上。本节解释 REPL 如何同时完成“切换内存状态”和“保留历史记录”。

避免：

> 会话管理是智能体系统中非常重要且复杂的一环。

### 4.2 给出一句准确结论

第二段直接回答本节问题。结论应包含主语、动作和结果。

> `/clear` 会创建新的 session id 和新的 `Soul`，不会删除旧会话文件。

### 4.3 展开输入、状态和输出

用短段落、步骤列表或表格说明：

- 输入来自哪里。
- 哪个对象接收输入。
- 哪些状态被读取或修改。
- 输出交给谁。
- 哪个条件会提前结束。

数据超过四项时用表格；有先后关系时用编号列表；只有两三句话时直接写段落。

### 4.4 映射真实代码

至少给出以下信息：

```text
入口：<函数或命令>
调用者：<谁触发它>
核心状态：<对象或字段>
返回值：<类型与含义>
失败方式：<异常、错误结果或降级>
源码：<准确路径>
```

只写文件名不够。读者还需要知道从哪里开始读，以及这个函数与上下游的关系。

### 4.5 用验证收尾

小节结尾给一个可执行检查，或一个明确观察点：

```bash
./.venv/bin/python -m pytest tests/test_session_store.py::test_clear_keeps_old_session -q
```

如果没有专门测试，就给临时目录实验，并明确它不是自动化测试。

## 5. 不同类型的小节怎么写

### 5.1 概念小节

适用于：Agent Loop、Tool、Hook、Skill、MCP、Compaction 等首次介绍。

内容顺序：

1. 没有这个概念时会出现什么具体问题。
2. 一句话定义。
3. 一个最小输入输出例子。
4. 它与前一章概念的关系。
5. 当前项目中的实现位置。

示例骨架：

```markdown
### Toolset 是什么

模型只会返回工具名和参数，不会替 Python 找到对应函数。没有统一分发层，每新增一个工具都要修改 Agent Loop。

Toolset 是工具注册与分发对象。它接收工具名和 JSON 参数，执行对应 Tool，返回统一结果。

输入：`("ReadFile", {"path": "README.md"})`

输出：`{"stdout": "...", "exit_code": 0, ...}`

正式实现位于 `src/whale_cli/soul/toolset.py`，由 `Soul.run()` 在处理 `tool_calls` 时调用。
```

### 5.2 流程小节

适用于：启动流程、一次模型调用、工具调用、压缩、MCP 发现、会话恢复。

内容顺序：

1. 明确触发点。
2. 按真实调用顺序编号。
3. 标出每一步读写的状态。
4. 写正常结束和失败结束。
5. 给出一条完整调用链。

推荐格式：

```text
触发：模型返回 tool_calls
  -> Soul.run() 读取工具名与参数
  -> Toolset.handle() 查找 Tool
  -> Approval 判断是否允许
  -> Tool.__call__() 执行
  -> 结果写入 role=tool 消息
  -> 下一轮 LLMClient.chat()
结束：模型返回不含 tool_calls 的 assistant 消息
```

流程图只在分支较多、纯文字难以追踪时使用。不要为了装饰增加图。

### 5.3 源码小节

适用于：解释一个类、函数或模块的真实实现。

代码块前必须写：

- 这段代码来自哪个文件。
- 为何只看这几行。
- 读者应该关注哪个状态变化。

代码块后必须写：

- 调用前状态。
- 执行中的关键分支。
- 调用后状态。
- 省略了哪些代码。

禁止事项：

- 贴出与仓库不一致的“近似源码”却不标注。
- 把伪代码放进 `python` 代码块。
- 逐行翻译明显语法，如“这一行定义变量”。
- 省略 import、依赖或入口后仍声称代码可独立运行。

### 5.4 设计原理小节

适用于：解释为何分层、为何保留状态、为何选择某种失败策略。

每个设计点回答四个问题：

1. 约束是什么。
2. 当前选择是什么。
3. 这个选择解决了什么。
4. 它带来什么代价。

示例：

> `Soul` 不直接用 `if/elif` 分发工具，而是交给 `Toolset`。约束是工具数量会增长，而且插件和 MCP 工具在启动时才确定。注册表让 Agent Loop 不依赖具体工具；代价是参数错误和执行错误都要在 Toolset 中统一转换，否则不同工具会产生不一致结果。

不要写“这样更优雅”“这样更灵活”就结束。必须说明具体灵活在哪里，以及代价是什么。

### 5.5 配置小节

适用于：LLM、agent.yaml、hooks、skills、plugins、MCP 配置。

内容顺序：

1. 默认配置路径。
2. 最小可运行配置。
3. 字段表：字段、类型、默认值、是否必填、错误表现。
4. 配置优先级。
5. 密钥和权限提醒。
6. 一条验证命令。

示例配置不能包含真实密钥。占位符要明确说明是否真的支持环境变量展开，不能只写一个看起来合理的 `${TOKEN}`。

### 5.6 错误与安全边界小节

适用于：审批、Bash、文件操作、远端请求、MCP、后台任务。

建议使用表格：

| 场景 | 当前行为 | 用户看到什么 | 应先检查什么 |
|---|---|---|---|
| 工具名不存在 | Toolset 返回错误结果 | `exit_code=1` | 模型返回的工具名 |
| 审批拒绝 | 不执行 Tool | `exit_code=126` | safe/yolo 模式 |

安全结论必须准确：审批不是沙箱；路径检查不等于系统权限隔离；外部 MCP server 仍是受信任代码或服务。

### 5.7 测试小节

测试小节不能只写“运行 pytest”。至少说明：

1. 精确命令。
2. 测试函数名。
3. 测试输入。
4. 核心断言。
5. 哪些真实风险没有覆盖。

推荐格式：

````markdown
## 本章测试

运行：

```bash
./.venv/bin/python -m pytest tests/test_mcp.py::test_soul_round_trip_calls_real_mcp_stdio -q
```

这个测试让 MockLLM 先选择 MCP echo 工具，再让真实 stdio server 返回结果。核心断言是 `role="tool"` 消息包含 `echo:agent-round-trip`，并且第二轮模型能给出最终回答。

它不验证外部 HTTP server 的稳定性，也不验证 OAuth。
````

### 5.8 动手实验小节

每章安排两个实验：

- 正常路径：输入正确，观察机制成功工作。
- 失败路径：故意给错参数、关闭依赖或触发限制，观察错误如何传递。

实验必须写清：准备条件、执行命令、预期现象、清理方式。不要把模型的完整回答写死，只断言稳定现象，如文件存在、工具被调用、状态改变或错误码出现。

### 5.9 小结小节

小结只回答以下内容：

- 本章新增了哪个机制。
- 它接入了哪条已有链路。
- 当前还有哪个明确边界。

不要重新复述整章，也不要用“未来可期”“迈出了重要一步”一类空结尾。

## 7. 可直接发给基础模型的任务指令

复制下面的指令，把占位符替换为当前章节信息。

````markdown
你要修改 Whale CLI 教程 `<教程路径>`。

对应源码：

- `<源码路径 1>`
- `<源码路径 2>`

对应测试：

- `<测试路径>`

本章只讲：`<核心机制>`。

前置概念：`<读者已经掌握的内容>`。

不要扩展到：`<本章明确不讲的内容>`。

执行要求：

1. 先完整读取教程、源码和测试。不得只根据旧正文改写。
2. 列出旧正文与当前代码不一致的事实，然后直接修正文档。
3. 使用 `docs/教程小节标准化写作模板.md` 的章节顺序。
4. 每个首次出现的名词写清定义、输入、状态、输出和实现位置。
5. 每个流程写清触发、调用顺序、状态变化、正常结束和失败结束。
6. 代码前说明阅读目的，代码后解释运行时状态变化。
7. “当前实现”“教学简化”“扩展练习”“生产差距”必须分开写。
8. 不得声称源码中不存在的功能已经实现。
9. 示例不得包含真实 API Key，不得把审批写成沙箱。
10. 给出精确 pytest 命令、测试函数名、测试输入和核心断言。
11. 保留章节原有有效内容；删除重复、宣传式和无法核对的段落。
12. 修改完成后运行对应测试、全量测试和 `git diff --check`。

写作风格：

- 用直接、自然的中文。
- 一段通常 2 至 5 句。
- 少用粗体、破折号、表情符号和口号。
- 不用“至关重要”“深入探讨”“这不仅是”等空泛表达。
- 不固定模型回答全文，只描述可稳定观察的行为。

交付时报告：

- 修改了哪些小节。
- 修正了哪些事实错误。
- 对应测试结果。
- 仍未解决的边界。
````

## 8. 基础模型分批修改策略

不要让基础模型一次重写整章。建议一轮只处理一种内容：

| 轮次 | 任务 | 验收方式 |
|---|---|---|
| 1 | 核对事实和源码路径 | 每个功能结论能找到源码或测试 |
| 2 | 补开场、目标和最小模型 | 初学者不读源码也能复述输入输出 |
| 3 | 补完整流程和真实实现 | 调用顺序与源码一致 |
| 4 | 补设计原理和生产差距 | 每个选择同时写收益和代价 |
| 5 | 补实验、测试和自测题 | 命令可运行，断言稳定 |
| 6 | 语言清理 | 删除重复、口号和 AI 腔 |

每轮只修改指定小节。下一轮开始前，重新读取已修改文件，避免覆盖前一轮结果。

## 9. 交付前检查表

### 事实

- [ ] 标题中的功能确实存在。
- [ ] 函数名、文件路径、配置字段与当前代码一致。
- [ ] 返回值、错误码、默认值和停止条件已核对。
- [ ] 当前实现与生产差距分开描述。
- [ ] 没有把审批、路径检查或提示词约束称为安全沙箱。

### 教学

- [ ] 开场是具体失败场景，不是抽象口号。
- [ ] 本章只新增一个主要机制。
- [ ] 首次出现的术语有白话定义和最小例子。
- [ ] 先讲最小模型，再讲真实代码。
- [ ] 流程包含输入、状态变化、输出和停止条件。
- [ ] 设计原理写了约束、选择、收益和代价。

### 代码

- [ ] 代码块语言标记正确。
- [ ] 伪代码没有标成 Python。
- [ ] 教学代码和正式实现的差异已经说明。
- [ ] 可运行示例包含依赖、import 和入口。
- [ ] 没有真实密钥、私人路径或不可复现输出。

### 验证

- [ ] 给出了精确测试文件和测试函数。
- [ ] 说明了核心断言，而不是只写“测试通过”。
- [ ] 正常路径和失败路径都能实验。
- [ ] 对应测试通过。
- [ ] 全量测试通过，或明确记录与本章无关的已有失败。
- [ ] `git diff --check` 通过。

### 语言

- [ ] 段落没有重复同一结论。
- [ ] 删除“至关重要”“值得注意”“未来可期”等空话。
- [ ] 没有连续使用机械的三项排比。
- [ ] 粗体、引用块和表格只在确实有助于理解时使用。
- [ ] 小结写的是结论和边界，不是重新列目录。

## 10. 评分标准

每章完成后按 100 分检查。低于 80 分不进入下一章。

| 维度 | 分值 | 扣分示例 |
|---|---:|---|
| 事实准确 | 30 | 声称不存在的功能已实现；路径或参数错误 |
| 学习顺序 | 20 | 未定义术语；直接贴复杂源码 |
| 代码映射 | 15 | 没有入口、调用者、状态和返回值 |
| 设计解释 | 10 | 只有“更灵活”，没有约束和代价 |
| 测试验证 | 15 | 只有命令，没有输入、断言和覆盖边界 |
| 语言质量 | 10 | 重复、宣传语、AI 腔、格式滥用 |

出现以下任一问题时直接退回修改，不看总分：

- 写入真实密钥。
- 编造源码、测试结果或外部引用。
- 把未实现能力写成当前实现。
- 删除已有正确内容，却没有等价替代。
- 示例命令会破坏用户数据，又没有警告和隔离环境。
