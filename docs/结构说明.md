# Whale CLI 架构说明

## 1. 项目定位

**Whale CLI** 是一个教学级 AI Agent CLI，保留工具注册表、ReAct 循环、TodoList、上下文压缩、权限审批等核心能力闭环，但代码极简、模块清晰、全程可测试。目标是让读者用最少的代码理解"一个能在终端完成任务的工程搭档"是怎么造出来的。

默认接入 Step Plan 的 `step-3.7-flash`（OpenAI 兼容），可切换任意兼容端点。

## 2. 目录结构

│   └── approval.py            # 权限审批

## 3. 分层职责

### 3.1 LLM 层（`llm/`）

- **`LLMClient`**：薄封装 OpenAI SDK。`chat(messages, tools)` 返回 SDK message 对象，支持 `max_tokens` / `tool_choice` / `temperature`。
- **配置三级优先**：环境变量（`STEP_API_KEY` 等）→ `~/.whale/config.json` → 交互输入。
- **`max_context_tokens`**：从配置读（step-3.7-flash = 256000），供压缩层判断阈值。

### 3.2 工具层（`tools/`）

- 每个工具继承 `Tool` 基类，声明 `name` / `description` / `schema` / `__call__(**kwargs)`。
- 返回值统一契约：`{stdout, stderr, exit_code, changed_files}`。
- 危险工具（Bash/WriteFile/Edit）设 `approval_action`，由 Toolset 在调用前过审批闸门。

### 3.3 灵魂层（`soul/`）—— 项目核心

- **`Soul`**（`soul.py`）：Agent 主循环。`run(user_input)` 执行 ReAct：每轮先（按需）压缩 → 调 LLM → 若有 `tool_calls` 则经 Toolset 分发执行、结果回填 → 否则输出最终回答。`max_steps=15` 防死循环。
- **`Toolset`**（`toolset.py`）：工具注册表。`register` / `get` / `all_schemas` / `handle(name, args_str)`。`handle` 内部完成 JSON 解析 → 审批闸门 → `tool(**args)` → 异常包装，永不让单次工具失败崩掉整个循环。
- **`TodoStore`**（`todo_store.py`）：内存任务列表，三态 `pending/in_progress/done`，整体替换语义。
- **compaction**（`compaction.py`）：`estimate_tokens`（字符/4）+ `should_compact`（0.85 比例）+ `compact`（保留 system + 最近 2 条 + 其余喂 LLM 生成结构化摘要）。
- **`Approval`**（`approval.py`）：三档 `approve / approve_for_session / reject`；`yolo` 模式全放行；`auto_approve_actions` 集合记忆本 session 已"不再询问"的动作。

### 3.4 存储层（`storage/`）

- **`SessionStore`**：jsonl 存储。`<base>/sessions/index.jsonl`（append-only，last-wins）+ `<base>/sessions/<session_id>.jsonl`（消息流）。
- `SessionInfo` dataclass：`session_id / title / created_at / updated_at`。

### 3.5 UI 层（`ui/`）

- **`shell/main.py`**：REPL。斜杠命令 `/help` `/exit` `/clear` `/session` `/todo` `/yolo` `/safe`；非 Windows 用 `input()`，Windows 用 `msvcrt` 支持 ESC 暂停 / TAB 补全。

## 4. 关键设计取舍

| 决策 | 选择 | 理由 |
|---|---|---|
| 工具 schema | 手写 JSON Schema（不引入 pydantic） | 降低新手门槛 |
| 工具返回值 | `{stdout,stderr,exit_code,changed_files}` | 与原始工具契约一致 |
| 工具分发 | `Toolset.handle` 注册表 | 新增工具不改 Soul |
| Todo | 单工具 `TodoWrite`，整体替换 | 保持整体替换语义，降低状态同步复杂度 |
| 压缩 | 字符/4 估算 + 0.85 比例 | 无 tokenizer 依赖，适合教学项目 |
| 审批 | action 字符串三档，同步阻塞 | 不做命令正则匹配，保留审批主干 |
| 模型 | step-3.7-flash（256K 上下文） | OpenAI 兼容，支持 tool calling |

## 6. 和生产级参考实现的结构映射

Whale CLI 不是要完整复刻生产级项目，而是把真实 CLI Agent 的 harness 机制拆成能教学、能验证、能继续补的最小结构。

| 能力 | Whale CLI 当前结构 | 生产级参考结构 | 教学版实现边界 |
|---|---|---|---|
| Agent loop | `soul/soul.py` | 独立 agent loop 模块 | 保持单循环清晰，逐步补事件点 |
| Agent 规格 | `agents/default/agent.yaml` + `system.md` | `agents/default/*.yaml`, `soul/agent.py` | 轻量 `AgentSpec`，不做完整 agentspec |
| Runtime 上下文 | `Soul.__init__` 里直接组装 | `Runtime` dataclass | 保留显式构造，避免引入大型 Runtime |
| Toolset | `soul/toolset.py` | `soul/toolset.py`, `tools/`, plugin/MCP | 保持统一 Tool 协议，给外部工具留 adapter |
| Approval | `soul/approval.py` | `approval_runtime/`, `soul/approval.py` | 保持同步审批，后续支持权限冒泡 |
| Hooks | `hooks/`, `Toolset` 触发 Pre/PostToolUse | `hooks/events.py`, `hooks/engine.py` | Python callback 版 HookEngine |
| Skills | `skill/`, `skills/whale-cli-help` | `skill/`, `skills/` | project/user/builtin 三层发现 |
| Project instructions | `context/project.py` | `load_agents_md()` | root→leaf 合并 `AGENTS.md`，带大小预算 |
| Subagents | `subagents/`, `tools/agent/` | `tools/agent/`, `subagents/` | 前台 subagent，独立 messages，返回 summary |
| Background tasks | `background/`, `tools/background/` | `background/` | 线程版后台 Bash，任务状态和输出落盘 |
| MCP / plugins | `plugin/loader.py` | `plugin/`, MCP config, `load_mcp_tools` | 项目本地 plugin 工具适配成 `Tool` |

进阶教程的 11-18 章就是按这张表展开：先解释生产级参考实现为什么这样做，再给出 Whale CLI 的教学版实现边界和验收方式。
