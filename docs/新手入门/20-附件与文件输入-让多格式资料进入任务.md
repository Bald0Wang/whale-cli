# 20. 附件与文件输入：让多格式资料进入任务

本章导航：

- 新增机制：浏览器附件选择、拖放、上传管理、文本摘要与任务上下文注入。
- 正式入口：`webui/src/App.jsx`、`webui/server.py`。
- 验证方式：上传一个 `.md` 或 `.csv`，发送“总结附件”，确认任务消息含有附件摘要。
- 本章不展开：OCR、扫描版 PDF 识别、视频/音频输入与长期文件库。

聊天输入框不应该只有一行文本。用户常常希望“看看这个需求文档”“总结这份 CSV”“解释这张架构截图”。

但把“允许选择文件”当成附件功能是不够的。真正的输入系统至少要回答：文件能不能上传、放在哪里、模型实际看到了什么、什么时候删除、哪些格式只是保存而不能解析。

本章把这些边界拆开。

## 20.1 先建立正确目标

当前 Whale CLI 的附件功能目标是：**把用户主动选择的资料，以受限且可解释的方式带进当前 Agent 任务。**

它不是网盘，也不是自动扫描整个电脑。附件必须由用户在输入框中选择或拖入，随后经过类型、大小和数量检查。

当前支持的后缀：

| 类别 | 支持格式 | 当前处理方式 |
|---|---|---|
| 纯文本与结构化文本 | `txt`、`md`、`json`、`csv` | UTF-8 读取，截取有限摘要后注入任务上下文 |
| 现代 Office 文档 | `docx`、`xlsx`、`pptx` | 使用 ZIP/XML 结构做有限文本提取 |
| PDF | `pdf` | 尽力提取可见文本片段，不保证扫描件或复杂排版效果 |
| 图片 | `png`、`jpg`、`jpeg` | 生成本地预览；开启视觉输入后按 OpenAI 兼容 `image_url` 发送给模型 |
| 旧格式 | `doc`、`xls`、`ppt` | 可上传、可管理，但不承诺文本提取 |

每个文件最大 24 MB，一次最多附加 8 个。图片超过 6 MB 或边长超过 2560px 时，浏览器会在上传前压缩为 JPEG。限制不是为了“故意不方便”，而是避免浏览器、会话上下文和模型上下文同时被大文件拖垮。

## 20.2 从加号到任务的一条路径

输入框的加号不再只是装饰。它打开原生文件选择器；用户也可以把文件拖到输入框上。

```text
用户选择/拖放文件
    -> 前端检查数量与大小
    -> 图片必要时缩放与压缩
    -> multipart/form-data POST /api/uploads
    -> AttachmentStore 校验、保存、提取摘要
    -> 输入框显示附件条，可移除
    -> POST /api/runs 携带 attachment_ids
    -> 服务端组合 prompt + 附件摘要
    -> Soul.run() 进入 Agent Loop
```

这里有一个很容易忽略的设计点：上传使用 `multipart/form-data`，不再把文件编码为 base64 JSON。这样避免了约三分之一的编码膨胀，也不会触发小 JSON 请求体限制。前端发送给 `/api/runs` 的不是文件内容，而是上传成功后拿到的 `attachment_ids`。文件只上传一次，任务请求只引用 ID。这能避免每次点击发送都重复传输大文件。

## 20.3 前端：把附件作为输入状态管理

对应代码：`webui/src/App.jsx` 的 `Composer` 与 `uploadFiles`。

输入状态至少包含：

```js
{
  id: "att_...",
  name: "requirements.md",
  extension: ".md",
  mime_type: "text/markdown",
  size: 2048,
  kind: "file",
  preview_url: ""
}
```

`Composer` 用隐藏的 `<input type="file" multiple>` 触发系统选择器，并在 `onDrop` 里接收拖放文件。上传成功后，附件条显示文件名、格式、大小；图片会使用 `preview_url` 显示缩略图。点击附件条上的关闭按钮会调用 `DELETE /api/uploads/{id}` 并从当前输入状态移除。

发送后，用户消息会把附件的轻量 metadata（ID、名称、类型、大小和图片预览 URL）一起写入会话记录。聊天气泡因此会直接显示图片缩略图，点击可查看原图；图片二进制和视觉请求中的 base64 不会写进 JSONL。附件存储同时保留一个小型索引，服务重启后仍能为尚未删除的图片恢复预览。

发送条件也需要变化。以前只有 `value.trim()` 非空才能发送；现在“没有文字但有附件”同样是一个合法任务。系统会补上默认指令：

```text
请分析我附加的文件。
```

这让“上传文件后直接请它总结”成为完整工作流，而不是要求用户再输入一段占位文本。

## 20.4 后端：AttachmentStore 不等于任意文件读取

对应代码：`webui/server.py` 中的 `AttachmentStore`。

上传接口只接受白名单中的扩展名。文件名会先经过 `Path(name).name` 处理，去掉用户给出的目录部分；实际文件用随机 `att_...` ID 保存到 Whale 的运行数据目录。浏览器不会把本地绝对路径传给服务端，也不会让 Agent 根据任意路径读取用户电脑。

最小存储结构如下：

```text
.whale_cli/
  uploads/
    att_a1b2c3d4e5f6.md
    att_d7e8f9a0b1c2.png
```

`AttachmentStore.payload()` 返回的元信息不包含这个磁盘路径，只提供附件 ID、名称、类型、大小、摘要和图片预览 URL。这样 UI 能管理附件，而实现细节不会变成浏览器接口的一部分。

## 20.5 为什么要做“有限摘要”

把完整文件直接拼进 prompt 看似最简单，实际很危险：

- 大 CSV 可能瞬间耗尽上下文预算。
- JSON、Office XML 或 PDF 内部数据常包含大量无意义结构。
- 用户会误以为模型已经完整阅读了被截断的文件。

因此当前实现执行两个限制：

1. 只从可提取格式中取文本。
2. 摘要最多保留 `MAX_ATTACHMENT_EXCERPT` 个字符，超出时明确标记“内容已截断”。

任务最终收到的上下文形态类似：

```markdown
## 本次任务附件
- `requirements.md` (text/markdown, 2048 bytes)
  已提取的内容摘要：
  # 登录改造
  - 支持邮箱验证码
  - 补充失败重试测试
```

这不是“自动总结”。它只是受限的原始文本摘录，模型仍需要根据用户目标判断该如何使用。

## 20.6 图片视觉输入与旧格式的诚实边界

Whale WebUI 的“API 与模型”设置现在提供**视觉输入**开关。对于支持视觉的模型，例如已配置的 `step-3.7-flash`，开启后会把当前任务中的 PNG/JPG/JPEG 图片转成 OpenAI 兼容的多段用户消息，并随同文字指令发出：

```json
[
  {"type": "text", "text": "请解释这张图"},
  {"type": "image_url", "image_url": {"url": "data:image/png;base64,..."}}
]
```

“图片细节”提供两个档位：`low` 适合快速识图并节省 token；`high` 更适合 OCR、图表和细节密集的截图。该字段会直接写进 `image_url.detail`。

视觉图片的 base64 只保留在当前 `Soul.run()` 的内存请求中。会话 JSONL 仍保存文字任务和附件说明，而不保存巨大的 data URL；因此历史会话可以回看“用了哪张图片”，但不会在重开后自动再次发送原图。

这个开关不能让不支持视觉的模型突然获得看图能力。切换模型或 Base URL 后，应先确认该供应商接受 OpenAI 兼容的 `image_url` multipart 内容。StepFun 的 Chat Completions 文档定义了该消息结构和 `low/high` 细节参数：<https://platform.stepfun.com/docs/zh/api-reference/chat/chat-completion-create>。

同样，老式 `doc/xls/ppt` 格式会被接受和保存，但当前不承诺内容提取。生产实现可以接入受控的格式转换服务或专用解析器，并为失败情况提供明确回退。

## 20.7 安全边界

本章的安全点不是“上传前问一次”而已：

- 扩展名白名单拒绝未知类型。
- 文件名去目录化，避免上传名携带路径。
- 单文件大小和附件数量受限。
- 文本摘要有长度上限。
- 浏览器只能按附件 ID 获取当前进程内登记过的文件。
- 工作区浏览器仍隐藏 `.whale_cli`，上传运行数据不会混进普通项目文件。

注意，上传内容本身仍可能包含提示注入、敏感数据或恶意文档文本。模型在附件摘要中看到的内容和用户输入一样，都应被视为**不可信数据**，不能覆盖系统规则或绕过工具审批。

## 20.8 如何验证

先启动 WebUI 后，在输入框中点击回形针，选择任意 `md`、`csv` 或 `json` 文件：

1. 附件条应出现文件名、类型与大小。
2. 点击 `X` 后，附件应从输入区消失。
3. 附件存在时，即使不输入文字，发送按钮也应可用。
4. 发送“总结附件”后，运行消息应包含“本次任务附件”和已提取的文本。

离线测试：

```bash
./.venv/bin/python -m pytest tests/test_webui_settings.py -q
```

该测试覆盖文本附件摘要、PDF 片段提取、上传文件落盘、删除和不支持类型的拒绝。

## 当前实现的边界

- 当前附件存储是 WebUI 进程内管理的临时运行数据；重启服务后，旧附件 ID 不再可用。
- 当前没有 OCR、扫描 PDF 识别和表格语义解析。
- 图片仅在“视觉输入”开启时作为视觉 token 发送；关闭时仍可上传、预览和记录元信息。
- 当前只为任务注入有限摘要，不提供跨会话的附件检索或知识库。
