# 27. 学习档案与社区反馈：把进步留下来

本章导航：

- 新增机制：关联知识与产出物的本地学习证据、能力报告和可审阅的贡献草稿。
- 正式入口：`learning/portfolio.py`、`tools/learning/learning_tools.py`。
- 验证方式：记录一次练习，确认报告包含证据但不自动提交 PR。
- 本章不展开：排行榜、自动评价、自动发布社区内容。

学过的东西如果只留在一次对话里，很难回顾，也很难向别人展示。学习档案把“完成过什么、理解到什么程度、下一步要改进什么”变成学习者可拥有的记录。

## 27.1 什么算学习证据

证据不是泛泛的“今天学了很多”，而是一件可以回答“我因此多会了什么、留下了什么、接下来做什么”的小产出：

- `note`：自己的概念解释。
- `exercise`：完成一次可复现练习。
- `project`：对项目做出的改动或演示。
- `reflection`：一次卡点、错误和修正。
- `contribution`：准备反馈给社区的补充草稿。

```json
{
  "action": "add_evidence",
  "kind": "exercise",
  "title": "完成工具调用练习",
  "detail": "调用本地只读工具，并写下参数、结果和一次失败后的修正。",
  "concepts": ["工具调用", "Agent Loop"],
  "outcome": "能解释工具结果为何需要回填到下一轮。",
  "artifact": "learning-wiki/concepts/agent-loop.md",
  "next_action": "为失败分支补一个测试。"
}
```

`concepts` 把一次项目实践接回知识图谱；`outcome` 描述能力或理解的变化；`artifact` 留下可回看的笔记、命令结果、路径或提交；`next_action` 防止档案成为只回顾、不继续的日志。它们都来自学习者实际提供的事实，未知时宁可留空。

## 27.2 生成可回顾报告

```json
{"action": "report"}
```

报告是 Markdown，包含：

```text
当前目标
知识地图与掌握度（只作为自评背景）
项目的学习价值、前置检查、能力产出和下一步
关联知识、能力变化、可回看产出与后续行动
可贡献给社区的本地草稿
```

它从 `.whale_cli/learning/state.json` 的事实生成，不依赖模型记忆，也不会捏造“已掌握”的成果。

### 在 WebUI 浏览

WebUI 左侧的“学习档案”直接读取同一份本地状态。它不是一个分数面板，而是三个浏览区：

- **项目陪学**：查看项目为什么值得做、进入前检查什么、可验证产出和下一步。
- **能力与证据**：查看每条记录连接了哪些知识点、留下了什么产出、能力发生什么变化。
- **社区贡献草稿**：查看待审阅的问题、建议补充和复现或资料；页面不会创建 issue 或 PR。

顶部“在对话中整理证据”会预填一条记录指令，用于补充缺失的事实；“刷新档案”只重新读取本地 JSON。完整 Markdown 学习档案放在页面底部按需展开，因此既可以快速扫描，也能回顾原始报告。

## 27.3 社区反馈的正确边界

学习者可能发现教程缺少前置知识、示例无法运行或某个概念解释不清。把它记录为 `contribution` 证据时，`detail` 说明问题，`concepts` 说明涉及知识，`outcome` 写建议补充，`artifact` 写复现步骤或资料。报告会把它排成可审阅草稿；但提交前必须人工检查。

```text
学习证据 -> 贡献草稿 -> 学习者审阅 -> 手动创建 issue 或 PR
```

“自动提交 PR”会让错误内容和凭据风险一起放大，因此当前实现明确不做。

## 27.4 情绪价值来自可见进步

这里不靠虚假的连续打卡或夸张评价提供反馈。一个可读的学习档案能让学习者看到：原来自己从不理解 Agent Loop，到完成了工具调用练习，再到留下可复现的产出并知道下一步要补什么。这种可追溯的能力变化比一句“你很棒”更能帮助长期学习。

## 27.5 实用案例：四周结束时生成一份能回看的学习记录

四周后，小陈完成了 Agent Loop 练习，也给项目 README 增加了一段运行说明。他先记录一条真实证据：

```text
把“给工具调用示例补充运行说明，并验证本地只读工具结果”记录为项目证据。
```

Whale 调用：

```json
{
  "action":"add_evidence",
  "kind":"project",
  "title":"补充工具调用示例说明",
  "detail":"阅读示例、验证只读工具输出，并补充运行前提。",
  "concepts":["Agent Loop", "工具调用"],
  "outcome":"能说明工具输出会如何影响下一轮决策。",
  "artifact":"README 运行说明与本地验证记录",
  "next_action":"为错误输入补一个可复现示例。"
}
```

然后小陈输入“生成我的学习档案”，对应 `{"action":"report"}`。报告会汇总他的目标、知识节点自评、项目的学习价值与下一步，以及这条证据关联的能力和产出。若他发现教程缺少“为什么要把工具结果回填”的解释，可以把问题记录为 `contribution` 证据；Whale 只生成可审阅的补充方向，小陈仍需自己检查并手动创建 issue 或 PR。

## 27.6 验证

```bash
./.venv/bin/python -m pytest tests/test_learning.py -q
```

测试记录一条带关联知识、能力变化、产出物和下一步的练习证据，检查 Markdown 报告完整保留它们，并确认报告不含自动提交行为。

## 当前边界

- 当前报告先返回 Markdown，不自动写入文件；学习者可以审阅后再决定是否保存。
- 没有公开榜单，也没有将学习者数据发送到社区。
- 后续可增加导出 PDF、WebUI 时间线和人工审核后的贡献模板，但仍应保留用户确认。

至此，Whale CLI 已有一条完整但透明的学习闭环：资料检索、档案、地图、路线、复习、项目与成长沉淀。
