# 23. 双链知识图谱：把学过的东西连起来

本章导航：

- 新增机制：主题节点、前置关系、LLM Wiki 结构大纲，以及 Obsidian 兼容学习 Wiki 导出。
- 正式入口：`learning/knowledge.py`、`learning/wiki.py`、`tools/learning/learning_tools.py`。
- 验证方式：建立“Python 基础 -> Agent Loop”，WebUI 自动显示本地图谱；再同步为带 `[[wikilink]]` 的 Markdown 页面。
- 本章不展开：自动抽取全文知识图谱、向量语义关系和外部 LLM Wiki 应用同步。

笔记列表能记录“我看过什么”，却很难回答“这个概念内部是怎样工作的”。学习者需要看到主题之间的关系，也需要把一个困难主题拆成可解释、可练习、可追问的知识结构。

## 23.1 图谱的最小形态

一个节点可以是概念、资料、项目或笔记：

```json
{"title": "Agent Loop", "kind": "concept", "mastery": 1}
```

一条边有来源、目标和关系：

```json
{"source": "python-基础", "target": "agent-loop", "relation": "prerequisite"}
```

这里约定：`source` 是前置概念，`target` 是后续概念。因此“Python 基础是 Agent Loop 的前置条件”只有一条边，不保存一份正向和一份反向副本。

## 23.2 双链如何工作

`KnowledgeMap.node()` 查询节点时计算两组视图：

- `links_to`：这个节点指向什么。
- `linked_from`：哪些节点指向它。

```text
Python 基础 -- prerequisite --> Agent Loop

查询 Python 基础：links_to = [Agent Loop]
查询 Agent Loop：linked_from = [Python 基础]
```

这比复制两份边可靠。修改关系时只改一次，反向链接会自然更新。

## 23.3 逐步建立地图

先写少量高价值概念，不要把课程目录一次性全导入：

```json
{"action": "add_node", "title": "Python 基础", "mastery": 3}
```

```json
{"action": "add_node", "title": "Agent Loop", "mastery": 0}
```

```json
{
  "action": "link",
  "source": "python-基础",
  "target": "agent-loop",
  "relation": "prerequisite"
}
```

`mastery` 的范围是 0 到 5。它是学习者自评，用来帮助路线排序、复习间隔和学习报告，不是考试分数。它不会显示在知识图谱或 Wiki 页面中，避免把“我感觉学会了多少”误画成“知识本身的结构”。

## 23.4 把一个主题拆成 LLM Wiki

`KnowledgeMap` 只负责登记主题和主题之间的关系。要让图谱真正支持学习决策，Whale 再为每个需要深入学习的主题保存一份 `outline`。这里不把“术语、误区、问题”误当成和主题同级的知识节点，而是先回答学习者最在意的五件事：

```text
主题
├── 学习定位：它在当前路线的哪一段，进入前需要什么
├── 为什么值得学：它替学习者解决哪个真实问题
├── 核心模型：概念、机制与边界如何组成可解释的理解
├── 学完能做什么：能独立完成哪些可验证的任务
├── 下一步行动：现在可以动手完成的一次小练习
└── 学习提醒：易错认知、待解问题、术语和资料证据
```

这不是让模型凭空发明一份百科。对话中先读主题和已给资料，再调用 `LearningWiki` 保存结构；不确定的结论写到 `questions`，来源只填学习者提供或已经验证的资料：

```json
{
  "action": "outline",
  "concept_id": "agent-loop",
  "positioning": "完成 Python 基础后，用它理解 Agent 如何组织一次任务。",
  "learning_value": "它让学习者知道模型回答如何变成可观察、可验证的任务过程。",
  "outcomes": ["解释观察如何回填上下文", "实现一个最小工具调用循环"],
  "definition": "让模型、工具与观察结果持续协作的控制循环。",
  "mechanism": "模型决定下一步；工具返回观察；观察回填到下一轮上下文。",
  "key_terms": ["工具调用", "观察", "上下文回填"],
  "practice": "让一个只读工具调用后，再根据输出给出回答。",
  "pitfalls": ["把一次工具调用误当作完整循环"],
  "questions": ["何时应该停止循环？"],
  "sources": ["Whale CLI 第 03 章"]
}
```

WebUI 先显示主题总览；节点摘要优先显示“为什么值得学”，连线显示它与其他主题的前置、关联或证据关系。点击 `Agent Loop` 后才显示这六张学习决策卡。这样一张全局图不会被几十个笔记标题淹没，又能在需要时回答“我为什么学它、学完有什么用、下一步做什么”。右侧点击“通过对话补全学习定位”会把同样的提示预填到聊天框，模型保存完成后，画布会在下一次刷新显示内容。

## 23.5 导出为 Obsidian 学习 Wiki

只有 JSON 图数据时，学习者很难在对话之外回顾。`ObsidianLearningWiki` 将现有 `KnowledgeMap` 导出为一个工作区内的标准 Markdown vault，默认路径是：

```text
learning-wiki/
├── .wiki-schema.md       # Wiki 结构规则
├── purpose.md            # 学习目标与使用方式
├── index.md              # 概念导航
├── log.md                # 每次同步记录
├── .whale-graph.json     # 随 Markdown 导出的可携带图谱缓存
└── concepts/
    └── agent-loop.md     # Obsidian 页面
```

每个概念页带 YAML frontmatter，并把图中的边写成 Obsidian `[[wikilink]]`：

```markdown
---
title: "Agent Loop"
type: concept
whale_id: agent-loop
generated_by: whale-cli
outline: llm-wiki
---

## 学习定位
完成 Python 基础后，用它理解 Agent 如何组织一次任务。

## 为什么值得学
它让学习者知道模型回答如何变成可观察、可验证的任务过程。

## 学完能做什么
- 实现一个最小工具调用循环。

## 核心模型
让模型、工具与观察结果持续协作的控制循环。模型决定下一步；工具返回观察；观察回填到下一轮上下文。

## 前置知识
- [[python-基础|Python 基础]]
```

这借鉴了 [nashsu/llm_wiki](https://github.com/nashsu/llm_wiki) 的“本地 Markdown、schema、index、log 与 wikilink”结构，也采用 [sdyckjq-lab/llm-wiki-skill](https://github.com/sdyckjq-lab/llm-wiki-skill) 的本地优先和双向链接思路。Whale 当前只实现学习图谱的轻量导出层，不会调用这两个项目的桌面程序、安装脚本或远程 API。

### 在 CLI 中初始化和同步

先询问状态：

```json
{}
```

对应工具是 `LearningWikiStatus`。要创建 vault：

```json
{"action":"init","vault_dir":"learning-wiki","title":"我的 Datawhale 学习 Wiki"}
```

知识地图新增或修改后，使用：

```json
{"action":"sync"}
```

`LearningWiki` 会写入多个 Markdown 文件和最小的 `.obsidian/app.json`，因此在 Safe 模式中会出现 `write learning wiki` 审批。同步成功后，这个目录已经是一个独立的 Obsidian vault；Obsidian 的 Graph View 会从页面中的 `[[...]]` 链接建立关系。

如果已安装 Obsidian 桌面应用，也可以在同步后请求：

```text
请用 Obsidian 打开我的学习 Wiki。
```

Whale 会调用 `OpenLearningWiki`，并在批准 `register and open Obsidian wiki` 后，将受管目录登记到 Obsidian 的 Vault 清单，再通过 `obsidian://open` 打开 `index.md` 或指定概念页。登记只新增当前 `learning-wiki/` 的路径，不修改已有 vault 的笔记或配置。

### 同步到已有 Obsidian vault

已有笔记库时，不能让模型猜路径。先给出真实路径，或在启动 Whale 的终端设置：

```bash
export OBSIDIAN_VAULT_PATH="$HOME/Documents/我的笔记库"
```

然后输入：

```text
把当前学习 Wiki 同步到我的 Obsidian vault。
```

Whale 调用 `SyncToObsidianVault`，并在批准 `write external Obsidian vault` 后，把生成内容镜像到已有 vault 的 `Whale Learning Wiki/` 子目录。它要求目标目录已经是 Obsidian vault（存在 `.obsidian/`），不会覆盖其中原有笔记，也不会把受管 Wiki 的 `.obsidian` 设置带入目标 vault。

### 自动收录对话：让过程也成为 Wiki

概念图谱只记录“学过什么”，却没有记录“为什么当时卡住、后来怎么想通”。因此 Wiki 还提供一个**默认关闭**的 `auto_capture` 开关。开启后，每轮 Agent **正常完成**时会生成一页对话笔记：

```text
learning-wiki/
└── conversations/
    ├── index.md                  # 已收录对话导航
    └── 20260713T...-xxxx.md      # 一轮对话
```

在 CLI 对话中可以明确请求：

```text
请开启自动收录对话到学习 Wiki。
```

对应工具调用是：

```json
{"action":"set_auto_capture","enabled":true}
```

关闭时使用：

```json
{"action":"set_auto_capture","enabled":false}
```

每页有 `type: conversation`、记录时间、会话 ID 和模型名，并只保留两个部分：`用户输入` 与 `Whale 最终回复`。工具调用原始输出、附件二进制内容、模型隐藏推理都不会进入 Wiki。中止、失败或达到循环上限的回合也不会被自动收录，避免把半成品误当结论。

这项开关只影响项目内的 `learning-wiki/`。即使已经设置了外部 Obsidian vault，也仍需显式调用 `SyncToObsidianVault` 才会镜像过去。

### 在 WebUI 中读取

WebUI 左侧新增“学习图谱”。它直接读取项目内 `.whale_cli/learning/state.json` 中的 `KnowledgeMap`，每约 2.5 秒静默刷新一次。因此 CLI、技能或对话刚写入节点和关系，网页不必等待 `LearningWiki sync` 就能看到同一张本地图谱。

画布使用 React Flow 渲染：总览把主题按关系分布在同一张可缩放、可平移的画布上；节点摘要显示它的学习价值。点击一个主题后，画布把它放在中心，并在周围展开“学习定位、为什么值得学、核心模型、学完能做什么、下一步行动、学习提醒”六类卡片。节点可以拖动重新排布，左下角提供缩放与自动适配视图，右下角的小地图用于在较大的知识网络中定位。这样，学习者先看主题之间的关系，再按需进入一个主题的学习决策，而不会把笔记目录误认为知识图谱。

每条连线也有语义：`prerequisite` 表示前置知识，`related` 表示一般关联，`evidence_for` 表示资料或实践依据，`contains` 表示主题包含的学习决策卡。掌握度仍只用于路线、复习和报告，不作为图谱节点的视觉标签；知识结构与学习进度是两件不同的事。

`LearningWiki sync` 的职责没有消失：它仍会把当前快照写入 `learning-wiki/`，供 Obsidian、外部 vault 镜像和版本管理使用；其中 `.whale-graph.json` 是随导出携带的缓存，而不再是 WebUI 的唯一数据源。顶部的“自动收录对话”开关会直接更新本地 `auto_capture` 状态，并显示当前已收录轮数；“刷新图谱”用于立即拉取，而不是等待下一次轮询。

## 23.6 实用案例：把“看不懂 Agent Loop”变成可处理的问题

小陈读项目时卡在 `Agent Loop`。他不应该只记一条“没看懂”，而是让 Whale 建立两个概念和一条可检查的依赖：

```text
请把 Python 基础记为掌握度 4，把 Agent Loop 记为掌握度 1，
并记录 Python 基础是 Agent Loop 的前置知识。
```

Whale 会依次调用：

```json
{"action":"add_node","title":"Python 基础","mastery":4}
```

```json
{"action":"add_node","title":"Agent Loop","mastery":1}
```

```json
{"action":"link","source":"python-基础","target":"agent-loop","relation":"prerequisite"}
```

随后让 Whale 查询 `{"action":"show","concept_id":"agent-loop"}`。结果中的 `linked_from` 应包含 `python-基础`；查询 `python-基础` 时，`links_to` 应指向 `agent-loop`。小陈因此能区分“Python 不够熟”与“需要练习 Agent Loop 本身”。

接着，小陈输入：

```text
请把当前 KnowledgeMap 同步成 Obsidian 学习 Wiki，并告诉我在 WebUI 哪里查看图谱。
```

小陈建立节点后，WebUI 的“学习图谱”会先出现 `Agent Loop` 主题及其价值摘要。小陈点击“通过对话补全学习定位”，Whale 保存 `outline` 后，画布会显示定位、价值、核心模型、能力产出、行动和提醒六张卡片；点击卡片可预览同一页 Obsidian Markdown。接着 Whale 调用 `LearningWikiStatus`，再调用 `LearningWiki({"action":"sync"})`；同意写入后，小陈可以在 `learning-wiki/concepts/agent-loop.md` 看到这些章节和 `[[python-基础|Python 基础]]`。若小陈已经有自己的 Obsidian vault，他再提供路径并使用 `SyncToObsidianVault`，生成内容会进入独立的 `Whale Learning Wiki/` 子目录。

## 23.7 验证

```bash
./.venv/bin/python -m pytest tests/test_learning.py tests/test_webui_settings.py -q
```

测试确认重复添加同一条边不会产生副本，目标节点能看到 `linked_from`；一个主题保存 LLM Wiki 大纲后会生成六个语义卡片与对应 Markdown 章节，图谱节点和页面不会泄露掌握度；Wiki 导出仍包含 frontmatter 与 wikilink，自动收录默认关闭、开启后会生成对话页与索引。

## 当前边界

- 当前由用户或 Agent 通过工具显式建图；`outline` 也需要基于对话、已读资料或可信来源生成，不会自动理解整篇 PDF。
- `related` 表示一般关联，不能替代严格前置关系。
- 图谱用于解释和规划，不声称是完整的知识真相；没有来源的内容应保留为待核验或待解问题，不代表已经完成资料溯源。
- 自动收录是用户主动开启的本地留存，不是无限制会话备份；它不会保存工具原始输出、附件内容或隐藏推理。
- 当前不自动同步 Obsidian 插件、外部 LLM Wiki 桌面应用或远端图谱服务；它只生成本地可携带的 vault，并在用户明确给出已有 vault 路径后进行受控镜像。
